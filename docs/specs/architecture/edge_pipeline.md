# Edge Pipeline Architecture

## System Overview
This diagram illustrates the high-throughput IoT data pipeline. The `AnomalyValidationService` acts as an edge filter, sitting between the local factory Kafka stream and the centralized Azure SQL database to drop normal readings and drastically reduce cloud ingress costs.

Exactly! You've got it. We have divided the data into two distinct lanes (Kafka topics) based on their "urgency" and "permanence."

Here is how the architecture is currently wired:

### **Topic 1: `kiln-sensor-readings` (The High-Volume Stream)**
- **Purpose**: This is the "firehose" for all raw telemetry data generated by your simulator.
- **Traffic**: Carries **100%** of all readings (NORMAL, WARNING, and CRITICAL).
- **Consumption**: The `AssetSensorKafkaListener` listens here and performs two tasks:
    1. **SSE Broadcast**: It immediately forwards **everything** it receives to your browser dashboard so the gauges stay live.
    2. **Audit Persistence**: It checks each reading; if it's marked as `CRITICAL`, it records a permanent copy in the Oracle `asset_sensor_readings` table.

### **Topic 2: `active-alarms` (The Notification Lane)**
- **Purpose**: Reserved for external high-priority alerting (e.g., triggering pager notifications or SMS) for **CRITICAL** anomalies only.
- **Traffic**: Contains **only** the readings that the validator has confirmed as `CRITICAL`.
- **Database Interaction**: This topic **does not** touch the database. It is strictly a "stateless" stream for downstream microservices to consume if they need to react to a problem without the overhead of querying the DB.

### **Summary of Persistence**
| Reading Status | Sent to Dashboard (SSE)? | Saved to Database (Audit)? | Forwarded to `active-alarms`? |
| :--- | :--- | :--- | :--- |
| **Normal** | ✅ Yes (Live Feed) | ❌ No | ❌ No |
| **Warning** | ✅ Yes (Live Feed) | ❌ No (DB Hygiene) | ❌ No |
| **Critical** | ✅ Yes (Live Feed) | ✅ **Yes (Permanent Record)** | ✅ Yes (External Alert) |
